# Personal Memory Engine: 构建真正懂你的AI记忆引擎

> **当AI不再每次从零认识你——三层渐进式记忆架构解析**
> 
> 开源地址: https://github.com/longongzi/personal-memory-engine

---

## 一、为什么需要记忆引擎？

OpenHuman 在2026年登顶GitHub Trending，它传达了一个关键信号：**AI助手的下一个战场是「记忆」**。

传统的AI对话系统没有任何持久记忆。每次对话都是独立的会话——你需要反复告诉AI你的背景、偏好、上下文。这就像每次见朋友都要重新自我介绍。

Personal Memory Engine 的设计目标：

- **让AI记住你** — 不再需要每次重复背景信息
- **自动积累** — 连接数据源后自动采集和组织信息
- **分层抽象** — 从琐碎片段到人格画像，层层提炼

## 二、核心架构：三层渐进式记忆

```
┌─────────────────────────────────────────────┐
│              L3 人格记忆 (Persona)            │
│  ▸ 综合画像、长期偏好、知识结构              │
│  ▸ 从L2蒸馏，保留最高价值                   │
├─────────────────────────────────────────────┤
│              L2 场景记忆 (Scene)             │
│  ▸ 按主题/时间段聚类摘要                     │
│  ▸ 3000字符窗口，自动评分                   │
├─────────────────────────────────────────────┤
│              L1 原子记忆 (Atomic)            │
│  ▸ 原始数据片段（邮件/聊天/代码/网页）       │
│  ▸ 连接器自动采集，TokenJuice压缩           │
└─────────────────────────────────────────────┘
         ↑ 蒸馏 (distillation)
         ↓ 回溯 (recall)
```

### L1: 原子记忆层 (ATOMIC)

最底层，存储原始数据片段。每条记忆有一个唯一ID，包含：

- **内容**：Markdown格式的片段
- **元数据**：来源类型（chat/email/code/web）、时间戳、访问频率
- **重要性评分**：基于 `recency×0.4 + frequency×0.3 + length×0.2 + source×0.1`

数据来源可以是邮件、聊天记录、代码提交、网页书签等。通过连接器框架（Connector Framework）自动采集。

### L2: 场景记忆层 (SCENE)

当L1数据积累到一定量后，引擎自动执行蒸馏（Distillation）：

1. **聚类**：按标签、来源、时间窗口分组
2. **摘要**：生成3000字符以内的场景摘要
3. **评分**：内部重要性阈值淘汰低价值记忆

L2记忆是「主题化的时间线」——比如「2026年5月的Python项目讨论」就是一个场景。

### L3: 人格记忆层 (PERSONA)

最高层，从L2进一步蒸馏，形成稳定的长期画像：

- 用户的职业、兴趣、技术栈偏好
- 常见行为模式和交流风格
- 持续积累的知识领域目录

L3不会频繁变化——它代表用户作为「人」的稳定性。

## 三、TokenJuice 智能压缩

记忆引擎面临的核心挑战是**信息密度**。原始数据充满噪音——HTML页面、重复格式、广告导航。TokenJuice专门解决这个问题：

```
原始HTML (100KB) 
    │ HTML→Markdown 转换（BeautifulSoup + Markdownify）
    ↓
Markdown文本 (~40KB)
    │ 去重连续行 + 折叠空白
    ↓
清洗后文本 (~35KB)
    │ 正则丢弃样板（广告/导航/页脚）
    ↓
最终记忆 (~30KB, 压缩率≈70%)
```

### 压缩策略矩阵

| 策略 | 压缩率贡献 | 说明 |
|------|-----------|------|
| HTML→Markdown | ~60% | 去除标签结构，保留语义 |
| 行去重 | ~8% | 去除重复行 |
| 空白折叠 | ~5% | 多空格→单空格 |
| 样板丢弃 | ~15% | 正则匹配移除导航/广告 |
| 截断 | ~2% | 超过50KB时截断 |

**中文优化**：中文字符按2 token估算，保留中文标点和段落结构。

## 四、重要性与蒸馏算法

每条记忆都有重要性评分，决定它在蒸馏时的存留权重：

```python
importance = recency * 0.4 + frequency * 0.3 + length * 0.2 + source_type * 0.1
```

- **recency（0.4）**: 越新的记忆权重越大，指数衰减
- **frequency（0.3）**: 访问越频繁越重要
- **length（0.2）**: 内容长度反映信息量
- **source_type（0.1）**: 不同来源的基础权重差异

蒸馏过程：

1. L1→L2：每10条L1记忆触发一次场景蒸馏，按标签聚类
2. L2→L3：每5条L2记忆触发一次人格蒸馏，保留最高分
3. 低分记忆被压缩为摘要节点，不再保留完整内容

## 五、数据源连接器框架

引擎内置了可扩展的连接器体系：

```
┌─────────────────────────────────────┐
│          PollingCoordinator          │
│   每20分钟轮询所有已注册连接器        │
├─────────────────────────────────────┤
│  ┌─────────┐ ┌────────┐ ┌────────┐  │
│  │ RSS     │ │GitHub  │ │Web     │  │
│  │Connect  │ │Connect │ │Scraper │  │
│  └─────────┘ └────────┘ └────────┘  │
│  ┌─────────┐ ┌────────┐            │
│  │WeChat   │ │Email   │  ...       │
│  │(Stub)   │ │(Stub)  │            │
│  └─────────┘ └────────┘            │
└─────────────────────────────────────┘
```

每个连接器实现三个核心接口：

- `fetch()` — 采集数据
- `validate()` — 验证配置有效性
- `start()/stop()` — 生命周期管理

OAuth2/API Key/Webhook三种接入方式，适配不同数据源。

## 六、REST API 设计

所有功能通过REST API暴露：

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/memories` | GET | 获取记忆列表，支持分页/搜索 |
| `/api/memories` | POST | 创建新记忆 |
| `/api/memories/{id}` | GET/DELETE | 获取/删除单条 |
| `/api/memories/tree` | GET | 获取三层记忆树 |
| `/api/memories/query` | GET | 按标签/来源/关键词过滤 |
| `/api/distill` | POST | 触发蒸馏 L1→L2→L3 |
| `/api/statistics` | GET | 系统统计 |
| `/api/compress` | POST | TokenJuice压缩测试 |
| `/ws` | WebSocket | 实时事件推送 |

WebSocket支持服务端主动推送——记忆更新、蒸馏完成、新数据到达时实时通知前端。

## 八、快速体验

cd personal-memory-engine

# 2. 安装

## 九、设计哲学与路线图

### 设计原则

1. **本地优先** — 数据存本地SQLite，敏感信息不离开设备
2. **渐进式** — 从L1到L3逐步抽象，不会丢失原始信息
3. **可扩展** — 连接器、压缩规则、评分算法都可配置

### 路线图

| 阶段 | 计划 |
|------|------|

| v1.0 | 插件系统、可视化编辑、一键备份恢复 |
